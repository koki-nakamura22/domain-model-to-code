# ADR-004: RDB（SQLite + SQLAlchemy）を採用する

## ステータス
Accepted

## コンテキスト
ホテル予約システムのデータ永続化に使用するDBを選定する。ローカルで完結することが条件。

## 決定
**RDB（SQLite）をデータベースに、SQLAlchemyをORMに採用する。**

## 理由

### RDB vs NoSQL
- 予約の整合性にACIDトランザクションが必須。仮予約の在庫確保・二重予約防止はトランザクション分離が重要
- NoSQLではこれをアプリケーション側で担保する必要があり、複雑さが増す
- データが本質的にリレーショナル（Hotel→RoomType→Room、Reservation→Guest等の関連が多い）

### SQLite
- ゼロコンフィグでローカル完結（ファイル1つ）
- 本プロジェクトはデモ目的であり、本番規模のスケーラビリティは不要
- オニオンアーキテクチャによりDB層は隠蔽されるため、将来PostgreSQL等への差し替えが容易

### SQLAlchemy
- リポジトリパターンとの相性が良い
- ドメインモデル（Pure Python）とDBモデル（SQLAlchemyモデル）を分離して定義できる
- Django ORMと異なり、フレームワークに依存しない

## 結果
- データ永続化はSQLite + SQLAlchemyで行う
- ドメインモデルとDBモデルは分離し、リポジトリ実装内でマッピングする
- DBの差し替えはリポジトリ実装の変更のみで対応可能とする
