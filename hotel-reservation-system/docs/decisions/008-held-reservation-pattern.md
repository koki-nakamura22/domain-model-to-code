# ADR-008: 仮予約（Held Reservation）パターンで在庫整合性を担保する

## ステータス
Accepted

## コンテキスト
予約作成から決済完了までの間に、同じ部屋タイプに対して複数のゲストが同時に予約操作を行う可能性がある。決済処理は外部システム（決済ゲートウェイ）に委譲するため、数秒〜数十秒の遅延が発生する。

### 「予約＝即確定」の問題
```
ゲストA: 空室確認(1室あり) → 決済開始 → (待ち) → 決済完了 → 予約確定
ゲストB:                      空室確認(1室あり) → 決済開始 → 決済完了 → ？？
```
- 決済中は在庫を押さえていないため、AとBの両方が決済成功しオーバーブッキングが発生する
- または後から片方を弾く仕組みが必要になり、ユーザー体験が悪化する

## 決定
**仮予約（Held Reservation）パターンを導入し、決済前に在庫を一時確保する。**

## 仕組み
```
ゲストA: 空室確認 → 仮予約(在庫-1, TTL=15分) → 決済 → 確定
ゲストB:              空室確認 → 在庫なし → 別の部屋タイプへ or 待ち
```

### ルール
- 仮予約作成時に在庫を一時的に確保する
- TTLは15分。15分以内に決済が完了しなければ自動失効し、在庫を戻す
- 決済完了で「仮予約」→「確定」に遷移
- 決済失敗でも即座に失効し、在庫を戻す

### ステータス遷移
```
仮予約(HELD) → 確定(CONFIRMED)     ← 決済完了
仮予約(HELD) → 失効(EXPIRED)       ← TTL超過 or 決済失敗
```

## 理由
- 在庫の整合性を決済前の段階で担保できる
- オーバーブッキングを防止しつつ、長時間の在庫ロックも防ぐ（TTLで自動解放）
- ドメインモデルとして「予約には仮と確定の2段階がある」が明示され、ビジネスルールが読み取りやすい

## 結果
- Reservationのステータスに `HELD` を追加
- 仮予約作成時にドメインイベント `ReservationHeld` を発行
- TTL管理はアプリケーションサービスがスケジューリングする
- 決済完了 → `ReservationConfirmed`、失敗/TTL超過 → `ReservationExpired` を発行
