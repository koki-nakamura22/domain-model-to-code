# ADR-009: 決済ゲートウェイをモック実装とする

## ステータス
Accepted

## コンテキスト
予約確定には決済処理が必要だが、Stripe等の実決済サービスとの連携は本プロジェクトのスコープ外である。一方、決済の成功/失敗がドメインの状態遷移に影響するため、決済のインターフェースと振る舞いは正しくモデリングする必要がある。

## 決定
**決済ゲートウェイはインターフェースを定義し、インフラ層でモック実装を提供する。**

## 実装方針

### ドメイン層
- `PaymentGateway` Protocolを定義（`process`, `refund` メソッド）
- `Payment` 集約が決済のライフサイクル（PENDING → COMPLETED/FAILED、REFUNDED）を管理

### インフラ層（モック実装）
- 常に成功を返すモック、一定確率で失敗するモック等を切り替え可能にする
- 実装例: `MockPaymentGateway` クラス

### 将来の実連携
- `StripePaymentGateway` 等をインフラ層に追加するだけで切り替え可能
- ドメイン層・アプリケーション層の変更は不要（オニオンアーキテクチャの恩恵）

## 理由
- 実決済連携は認証情報管理・Webhook処理等が重く、ドメインモデリングの検証という本来の目的から逸れる
- ただし決済のインターフェースと状態遷移は正しく設計しておくことで、将来の実連携が容易になる
- モック実装により、テスト時の決済成功/失敗シナリオを制御できる

## 結果
- `PaymentGateway` Protocolをドメイン層に配置
- `MockPaymentGateway` をインフラ層に実装
- 決済成功/失敗の振る舞いはドメインイベント（PaymentCompleted / PaymentFailed）を通じて予約の状態遷移に反映される
