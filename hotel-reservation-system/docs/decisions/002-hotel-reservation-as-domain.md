# ADR-002: 題材としてホテル予約システムを採用する

## ステータス
Accepted

## コンテキスト
ドメインモデル図→コード生成のアプローチが実践で使えるか見極めるためのデモプロジェクトを構築したい。題材となるドメインとして以下の候補を検討した：

- ECサイト（オンラインショッピング）
- タスク管理 / プロジェクト管理
- レストラン予約
- 図書館システム
- SNS / マイクロブログ
- **ホテル予約システム**

## 決定
**ホテル予約システムを題材として採用する。**

## 理由

### 十分な複雑さがある
- 部屋タイプ、料金プラン、季節変動、連泊、キャンセルポリシーなど多くの概念が登場する
- 時間軸での不変条件（二重予約防止、チェックイン/チェックアウト）がドメインルールとして複雑
- DDDの典型パターン（集約、値オブジェクト、ドメインイベント、仕様パターン等）が自然に登場する

### 実践評価に適している
- 「モデルがコードにどう落ちるか」の腕試しとして最適な難易度
- 単純すぎず（会議室予約）、発散しすぎない（SNS）バランス
- 実際のビジネスとしてイメージしやすく、要件が明確に定義できる

### 深掘りの余地がある
- 集約の境界設計で議論が生まれる（予約と料金計算の分離等）
- ドメインイベント（予約確定、キャンセル、チェックイン等）が豊富
- 結果整合性やCQRSなど高度なパターンの適用も検討できる

## 結果
- ホテル予約システムのドメインを要件整理し、ドメインモデル図を作成する
- モデルの深さは制限せず、集約・値オブジェクト・ドメインイベント等をとことん設計する
