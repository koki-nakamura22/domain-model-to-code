# ADR-007: 管理者機能をAPI経由で提供する

## ステータス
Accepted

## コンテキスト
管理者がホテル・予約・部屋の状態を管理する機能が必要。Django Adminのような管理画面フレームワークを使うか、APIとして提供するかを検討した。

## 決定
**管理者機能はFastAPIの管理用エンドポイントとして実装する。管理画面フレームワークは使用しない。**

## 理由

### ドメインロジックの一貫性
- Django AdminなどのDB直接操作型の管理画面は、ドメインロジックをバイパスする危険がある
- 例: 予約ステータスを直接DBで変更すると、キャンセル料計算・在庫解放・ドメインイベント発行がスキップされる
- API経由であれば、管理者操作もゲスト操作と同じユースケース層を通り、ドメインルールが必ず適用される

### 管理者向けエンドポイントの例
- `GET /admin/reservations` — 予約一覧・検索（ホテル別、日付別、ステータス別）
- `PUT /admin/reservations/{id}` — 管理者による予約変更（ゲスト代行）
- `POST /admin/reservations/{id}/cancel` — 管理者によるキャンセル（ゲスト代行）
- `PUT /admin/rooms/{id}/status` — 部屋状態の変更（清掃完了→空室等）
- `POST /admin/hotels` — ホテル・部屋タイプ・料金プランのマスタ登録

### フロントエンドについて
- 本プロジェクトでは管理画面のフロントエンドは作成しない（APIのみ）
- Swagger UI（FastAPI自動生成）で操作確認が可能
- 将来的に管理画面が必要になった場合は、このAPIを呼ぶフロントエンドを別途構築する

## 結果
- 管理者機能は `/admin/*` エンドポイント群として実装する
- すべての操作はアプリケーション層のユースケースを経由する
- 認証・認可は将来の拡張領域とし、初期実装では省略する
