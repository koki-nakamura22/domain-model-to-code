# ADR-005: オニオンアーキテクチャを採用する

## ステータス
Accepted

## コンテキスト
ドメインモデル図からコードを生成するデモプロジェクトであり、ドメイン層の純粋さが重要。また、DBやWebフレームワークの変更が業務ロジックに影響しない構造が求められている。

## 決定
**オニオンアーキテクチャを採用する。**

## レイヤー構成

```
domain/           ← 最内層。Pure Python。外部依存なし
  models/         ← 集約、エンティティ、値オブジェクト
  events/         ← ドメインイベント
  repositories/   ← リポジトリのインターフェース（Protocol）
  services/       ← ドメインサービス

application/      ← ユースケース層。ドメイン層のみに依存
  use_cases/      ← ユースケース（コマンド系）
  queries/        ← クエリ（読み取り系）
  services/       ← アプリケーションサービス

infrastructure/   ← インフラ層。ドメイン層のインターフェースを実装
  persistence/    ← SQLAlchemy モデル + リポジトリ実装
  payment/        ← 決済ゲートウェイ（モック実装）

presentation/     ← 最外層。FastAPI ルーター
  api/            ← ゲスト向け・管理者向けエンドポイント
```

## 依存方向
- 依存は常に外側から内側へ向かう
- domain は何にも依存しない
- application は domain のみに依存する
- infrastructure は domain のインターフェースを実装する（依存性逆転）
- presentation は application を呼び出す

## 理由

### ドメイン層の保護
- ドメインモデルがPure Pythonで記述されるため、ドメインモデル図との対応が明確
- DBやフレームワークの都合がドメインロジックに侵入しない

### テスタビリティ
- ドメイン層は外部依存がないため、単体テストが容易
- リポジトリはProtocolで定義するため、テスト時にインメモリ実装に差し替え可能

### DB差し替えの容易さ
- リポジトリのインターフェースはドメイン層に、実装はインフラ層に配置
- SQLite → PostgreSQL等の移行はインフラ層の変更のみで完結

## 結果
- 上記のレイヤー構成でプロジェクトを構築する
- 各レイヤー間の依存違反はpyrightの型チェックとimportルールで検出する
